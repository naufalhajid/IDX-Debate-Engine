# IDX Fundamental Analysis

An asynchronous research and decision-support pipeline for swing-trade analysis on the Indonesian Stock Exchange (IDX). This system automates the transition from quantitative screening to deep qualitative auditing, using a multi-agent debate architecture to mitigate cognitive bias and structural financial controls to enforce risk parameters.

This software is built for research, execution tracking, and decision validation. It is not financial advice.

---

## Architecture Overview

The system operates as a sequential pipeline:

```text
[Quant Filter] -> [Regime Detector] -> [Data Scouts] -> [Debate Chamber] -> [CIO Judge] -> [Risk & Sizing] -> [Reports]
```

1. **Quant Filter**: Screens the market to build a candidates list using fundamentals and technical indicators.
2. **Regime Detector**: Uses realized volatility of the IHSG index (`^JKSE`) to dynamically adapt pipeline risk thresholds.
3. **Data Scouts**: Fetches and prepares fundamental, chart, and news data in parallel.
4. **Debate Chamber**: Coordinates bull and bear agents through iterative debate rounds to identify blind spots.
5. **CIO Judge**: Reconciles debate arguments using a conflict resolution matrix and outputs structured verdicts.
6. **Risk & Sizing**: Validates price action, snaps targets to exchange tick sizes, and applies sector diversification rules.
7. **Reports**: Writes auditable JSON runs and formatted Markdown summaries.

## Technical Highlights

### 1. Multi-Agent Debate Chamber
To prevent the positive-bias common in single-prompt LLM analysis, the debate engine (`services/debate_chamber.py`) models an institutional investment committee:
* **Scout Parallelization**: Fundamental, technical, and sentiment data extraction tasks run concurrently using cheaper, high-speed LLM models (`gemini-3.1-flash-lite`).
* **Anti-Groupthink Protocol**: The debate runs for up to 3 rounds. In Round 2, the bearish agent is programmatically restricted from repeating data from Round 1 and must actively challenge the bullish thesis's margin of safety.
* **Devil's Advocate**: If the agents reach a consensus too early (e.g., Round 1), a contrarian node is spawned to stress-test the agreement.
* **CIO Verdict Synthesis**: A final judge synthesizes the transcript using a high-reasoning model (`gemini-3.1-pro-preview`), applying strict schema validation (`schemas.debate.CIOVerdict`) to output targets, catalysts, and confidence scores.

### 2. Adaptive Planning and Resilience (`AdaptivePlanner`)
External scrapers, database interactions, and API connections are inherently prone to failures. Rather than failing the entire batch, the system integrates a structured failure taxonomy (`core/failure_taxonomy.py`) and a dynamic recovery planner (`core/adaptive_planner.py`):
* **Error Classification**: Normalizes exceptions into categorized categories (DNS, Quota, Authentication, Schema, Timeout).
* **Graceful Degradation**: If non-critical data (like news sentiment) fails to fetch, the planner allows the pipeline to proceed with a `PROCEED_PARTIAL` instruction but applies a **15% confidence penalty** to the final trade conviction score.
* **Fail-Safe Abort**: If a critical API quota or authentication error occurs, or if more than 5 tickers fail in a single run, the engine aborts the batch to prevent useless API billings.

### 3. Financial Guardrails and Market Compliance
The engine enforces structural exchange rules and risk parameters directly in Python code:
* **Tick Size Snapping**: Indonesian stock exchange rules dictate precise pricing tick sizes (fractions of Rp 1, Rp 2, Rp 5, Rp 10, or Rp 25 depending on the price range). Target prices generated by the LLM are snapped to valid tick boundaries.
* **Realized Volatility Regime**: Because the Indonesian market lacks a public VIX, the system calculates the 20-day realized volatility of ^JKSE. Under high volatility, it automatically lowers selection sizes, tightens risk/reward caps, and increases minimum conviction requirements.
* **Sector Cap Sizing**: Limits capital allocation to prevent over-concentration in a single sector, scaling position sizing based on real-time CIO confidence.

## Directory Structure

```text
├── core/                  # System-level config, regime detection, telemetry, and planners
├── db/                    # SQLAlchemy models and async session management
├── providers/             # Scraping and API adapters (yfinance, Stockbit, IDX crawler)
├── repositories/          # Async data access repositories (CRUD abstractions)
├── schemas/               # Pydantic v2 validation models and type definitions
├── services/              # LangGraph debate state machine and valuation tools
├── utils/                 # Logging configurations, helper functions, and mathematical models
├── tests/                 # Comprehensive Pytest suite
└── output/                # Generated reports and logs (ignored by git)
```

## Setup & Installation

### Prerequisites
* Python 3.12+
* `uv` (Fast Python package installer)
* Chrome/Chromium (Required for IDX/Stockbit web scraping)
* Gemini API Key (For debate and reasoning steps)

### Installation
Clone the repository and install the dependencies:
```bash
uv sync
```

Configure local environment settings:
```bash
cp .env.example .env
```
Edit the `.env` file to add your API credentials and override default settings.

---

## Execution Commands

### Run Full Analysis Pipeline
Run the quantitative candidate filter, launch debates, score, and write final reports:
```bash
uv run python orchestrator.py
```

### Run in Dry-Run Mode
Execute the orchestration pipeline using mock LLM responses to test system components and code paths:
```bash
uv run python orchestrator.py --dry-run --no-interactive --output-dir tmp/orchestrator_dry_run
```

### Run Isolated Debates
Run the LangGraph debate chamber for specific tickers:
```bash
uv run python run_debate.py --tickers BBRI BBCA --output-dir output/debates
```

### Regenerate Quantitative Candidates
Manually trigger the quantitative screener:
```bash
uv run python run_quant_filter.py
```
